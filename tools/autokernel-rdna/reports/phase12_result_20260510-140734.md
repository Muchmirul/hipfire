## Root Cause Analysis

`gemv_hfq4g256` on gfx1201 is **memory-bandwidth-bound**. Each Qwen3.5-9B
decode step transfers approximately:

- 4B MQ4 weights × ~7B parameters = ~3.5 GB
- At 93 tok/s and GDDR6 bandwidth 288 GB/s:
  - 3.5 GB / 93 tok/s = 37.6 ms/token
  - Measured avg_ms = 10.57 ms → ~28% BW utilization (other kernels share BW)

For `gemv_hfq4g256` specifically: each group is 136 bytes, 8 groups =
1088 bytes per iteration. With 32 waves × 4 SIMDs = 128 active waves,
each CU handles ~8 rows, issuing 1088/128 ≈ 8.5 bytes/cycle per row.

**Why 8x doesn't beat 4x:**  
With 4x unroll, 4 groups × 136 bytes = 544 bytes per wave per iteration.
The wave's 32 threads each access 4 bytes of nibble data (1 uint32 load),
so each wave issues 32 loads/iteration. At 4x vs 8x, the L2 pipeline is
already fully pipelined for this stride pattern — doubling the register
pressure (8 vs 4 accumulators, 8 vs 4 scale/zero/pk register sets) does not
yield additional bandwidth throughput, it merely increases VGPR pressure.

**Implication:** Any kernel where the bottleneck is GDDR6 stream bandwidth
(not arithmetic or L1/L2 latency) will not benefit from higher unroll. The
correct next target is a kernel where arithmetic overhead exceeds memory
overhead, or where the access pattern allows LDS-based tiling.

Given the bandwidth-ceiling insight, the highest-value next experiment is
**`attention_flash_asym3_tile`** (rank 4, 12% decode) — it has an
arithmetic component (attention score computation) that may benefit from
RDNA4-specific tuning, and it has never had a gfx12 variant.

Alternatively, **`fused_qkv_hfq4g256`** (rank 6, 8%) could reduce the 3×
launch overhead for Q/K/V projections and improve cache locality by keeping
activations in L1/L2 across fused groups.
