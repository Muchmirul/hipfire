### Experiment #1: gemv_hfq4g256_residual — REVERT

**Hypothesis:** Create a `gemv_hfq4g256_residual.gfx12.hip` variant with
`__builtin_amdgcn_s_prefetch_data` 2 quads ahead to hide L2 latency.

**Finding:** The base kernel `gemv_hfq4g256_residual.hip` already uses 4x group
unroll + packed uint32 loads. Adding `s_prefetch_data` added overhead that
exceeded the latency-hide benefit for this kernel at Qwen3.5-9B matrix sizes
(K=4096 → 16 groups/row → 4 quads → only 4 main-loop iterations).

| Metric | Baseline | Candidate | Speedup |
|--------|----------|-----------|---------|
| Decode tok/s | 93.2 | 91.1 | **0.977x ❌** |
| Prefill tok/s | 949 | 944 | 0.995x |

**Decision: REVERT.** Threshold = 1.08x. 0.977x is a 2.3% regression.

**Root cause:** With only 4 loop iterations, the `s_prefetch_data` instruction
overhead (~2-3 cycles per call × 2 calls/iteration = 4-6 cycles) exceeds the
memory latency amortized over 4 iterations. The hardware prefetcher already
handles this access pattern effectively at gfx1201's HBM3e bandwidth.

**Artifact retained:** `kernels/src/gemv_hfq4g256_residual.gfx12.hip` preserved
as documented negative result. `kernels.rs` dispatch reverted to base kernel.

### Findings for Future Experiments

1. **s_prefetch_data** is counterproductive for short weight rows (≤16 groups/row).
   Reserve it for large matrices (e.g., 27B model: K=7168 → 28 groups → 7 quads —
   prefetch would fire 5×, which may be more effective).

2. **gemv_hfq4g256 (rank 1)** already has a gfx1201 variant. Verify it is actually
   being dispatched (confirmed via `gemv_hfq4g256_for_arch` which maps gfx1201).
   Potential optimization: try 4x group unroll (vs current 2x) for 27B models.

3. **attention_flash_asym3_tile (rank 4)**: No gfx12 variant. Highest-impact
   remaining opportunity. Creating a wave32 gfx12 FA variant is the next priority.

4. **gemv_hfq4g256_residual on 27B**: With K=7168, prefetch may actually help.
   Create `gemv_hfq4g256_residual.gfx12.hip` targeting 27B model sizes as next
   experiment.
